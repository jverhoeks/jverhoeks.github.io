Supply Chain Attacks Hit Every Ecosystem. Here’s How to Defend Yours.

Supply chain attacks on package registries are no longer an npm problem. In March 2026, axios was compromised via a stolen maintainer token — two poisoned versions published, a cross-platform RAT phoning home within two seconds of npm install. The same attack pattern has hit PyPI, RubyGems, and Maven Central. The install step is the attack surface, and it exists in every language.

This post covers three independent layers of defense. Stop at any layer — each one meaningfully reduces your exposure. Add the next when you’re ready.

The three allow-* flags are new in npm 11.15 and close a real attack surface: non-registry sources (local tarballs, remote URLs, local directories) are a common dependency confusion and CI injection vector. Layer 2 — Harden your projects

Config committed to the repo means every developer, every CI run, and every environment gets the same policy — even before they’ve set up their machine.

The two controls that matter

1. Minimum release age. Most supply chain attacks follow the same pattern: compromise a package, publish a malicious version, hope developers install it before detection. A quarantine window breaks this. 7 days is long enough for the community to notice and for registries to yank the version.

2. Block install scripts. postinstall hooks run arbitrary code the moment you install a package — before you even import it. Disabling them globally, with explicit opt-in for the handful of packages that legitimately need them, eliminates this vector entirely.

Layer 3 — Org-wide enforcement

Client-side flags protect individual machines. A proxy enforces policy at a single chokepoint — every install from every developer, every CI runner, and every environment goes through it. It is also the only option for ecosystems with no native age gate: Go, Cargo, Maven, NuGet, and Composer cannot quarantine packages without a proxy.

escrow — recommended (7 ecosystems, age + OSV)

escrow v1.5.0 is a purpose-built supply-chain proxy covering seven ecosystems (npm, PyPI, Go, Cargo, NuGet, Maven, Composer) in a single binary. It enforces age gates, OSV vulnerability checks, and publisher-account-age policy server-side, with a real-time dashboard for reviewing and approving blocked packages.

Point each tool at port 7888: registry=http://localhost:7888/ for npm/pnpm/bun, index-url = http://localhost:7888/pypi/simple/ for pip/uv, GOPROXY=http://localhost:7888/go,off for Go, and so on. See the full repo README for every ecosystem’s routing config.

JFrog Curation — commercial, if you’re on the JFrog platform

JFrog Curation is the commercial equivalent. It intercepts package requests in front of Artifactory’s remote repositories and applies policy before packages enter your SDLC.

The key feature is the time-delay / immature package policy: blocks newly published versions until they reach a configurable minimum age (14 days cited as example). When a package is blocked, Curation can silently substitute a safe older version — useful for CI pipelines that can’t tolerate hard failures, but it reduces visibility compared to escrow’s explicit approval dashboard.

Things Most Teams Miss

ignore-scripts=true does not block git dependencies. npm calls the system git binary to fetch git-hosted packages, bypassing the lifecycle hook system. A malicious package can override which binary npm treats as git. The fix: allow-git=false (npm >= 11.10).

pnpm v11 moved build policy to pnpm-workspace.yaml. allowBuilds in .npmrc is silently ignored in v11. If you upgraded from v10, your build policy may be doing nothing.

npm uses days, not minutes. min-release-age=7 = 7 days in npm. minimumReleaseAge=7 = 7 minutes in pnpm. Both the key and the unit differ.

GOPROXY=...,direct is not safe. direct silently bypasses the proxy when it’s unreachable. Use off — fail loudly instead.

Maven checksumPolicy defaults to warn. A tampered artifact produces a warning in the build log and then installs. Set checksumPolicy=fail.

Composer scripts run exactly like npm postinstall — but fewer teams know this. Always use --no-scripts in CI. There is no persistent config key to disable them globally.

.npmrc flags don’t reach IDE-bundled runtimes. VSCode, Cursor, and Zed bundle their own Node.js. That runtime may not read ~/.npmrc. Use launchctl setenv (macOS) or /etc/profile.d/ (Linux) to inject NPM_CONFIG_* env vars at the OS level.

A proxy does not replace client-side script blocking. Every proxy — escrow, Nexus, Verdaccio — caches packages but does not strip postinstall hooks. The ignore-scripts, allowBuilds, and only-binary settings from Layer 2 are still required even when a proxy is in the path.

Cargo build.rs has no global off switch. cargo vet and cargo deny are the only mitigations short of sandboxed builds.

What This Does Not Cover

  • Dependency confusion: scoped packages (@myorg/), single index only, private proxy as sole source
  • Typosquatting: lockfiles are the main mitigation
  • SBOM generation: mvn cyclonedx:makeAggregateBom, cargo cyclonedx, dotnet sbom-tool
  • Lockfile integrity in CI: npm ci, pnpm install --frozen-lockfile, uv sync --frozen, cargo build --locked
  • Vulnerability scanning: npm audit, pip-audit, cargo audit, govulncheck
  • SLSA / Sigstore: provenance attestation — see cross-ecosystem/slsa-sigstore.md
